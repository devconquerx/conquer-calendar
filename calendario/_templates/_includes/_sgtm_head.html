{% comment %} Google Tag Manager — loader first-party (server-side GTM / sGTM). Réplica del snippet que sirve Webflow: carga el contenedor web de la marca a través del dominio first-party de sGTM (load.somos..com) con el path "cloaked", en vez de www.googletagmanager.com. El contenedor dispara Meta/GA4/TikTok hacia el server-side GTM. La config viene de funnels.context_processors.get_gtm_config(escuela): gtm.id → ID completo del contenedor web (GTM-XXXXXXX) — para el noscript gtm.st → ID sin el prefijo "GTM-" — lo que espera el loader (?st=) gtm.loader_domain → dominio first-party del sGTM gtm.loader_path → nombre del archivo cloaked (p.ej. tiehtchn.js) CARGA DIFERIDA: el contenedor de GTM (que a su vez baja GA/Cookiebot) NO se inyecta de forma síncrona en el —ahí compite con el render inicial y hunde el FCP/LCP en mobile—. Inicializamos `dataLayer` y empujamos `gtm.start` de inmediato (barato, sin red), pero la inyección del script del contenedor se difiere al primer gesto del usuario o, como respaldo, a unos segundos en idle. `dataLayer` es un array: cualquier evento que la app empuje antes (event_id, form_submit_lead, etc.) queda encolado y GTM lo procesa al cargar, así que NO se pierden conversiones. El submit del form siempre ocurre tras interactuar, o sea con GTM ya cargado. Para quitar el diferido, basta volver al snippet síncrono de siempre.
